# Maintainer Firewall V2 产品级模块蓝图（P0/P1/P2）

**目标：** 在现有 V1（MVP）稳定可用的基础上，补齐“产品级平台”能力，形成可扩展、可治理、可运营的维护者自动化系统。

**范围：** 本文只定义模块规划与优先级，不涉及具体代码实现细节。

---

## 1. 完整性判定标准（先定义“完整”）

一个“产品级”维护自动化系统，至少要形成 6 个闭环：

1. 接入闭环：事件可接、可验、可追溯。
2. 决策闭环：规则可配置、可版本化、可回放验证。
3. 执行闭环：动作可编排、可重试、可补偿。
4. 安全闭环：租户隔离、权限控制、审计留痕。
5. 运维闭环：SLO、告警、容量与故障恢复。
6. 生态闭环：多平台集成、对外 API/SDK、扩展机制。

结论：V1 已覆盖前 3 项的基础形态，但后 3 项仍不完整。

---

## 2. V1 现状到 V2 的核心缺口

1. 缺少多租户与组织级隔离（当前更偏单实例管理员模型）。
2. 权限控制粒度不足（缺少角色-资源-动作级授权矩阵）。
3. 规则生命周期缺失（版本、灰度、回滚、仿真回放）。
4. 执行层缺少可视化编排（多步骤流程、分支、审批）。
5. 运维治理不成体系（SLO、告警策略、容量基线、灾备演练）。
6. 平台扩展能力不足（多平台连接器、OpenAPI/SDK、插件机制）。

---

## 3. V2 模块规划（优先级）

## 3.1 P0（必须，产品化基线）

### P0-1 租户与资源边界模块

1. 目标：实现组织/项目/仓库级隔离与配额。
2. 核心能力：`tenant`、`project`、`environment` 三级模型；数据行级隔离；限流与配额。
3. DoD：任一租户无法读取其他租户规则、事件、告警、审计数据。

### P0-2 权限与安全治理模块

1. 目标：从“登录保护”升级为“细粒度授权”。
2. 核心能力：RBAC（角色-资源-动作）；高风险操作二次确认；审计不可篡改链路。
3. DoD：所有受保护接口均可映射到统一授权策略并留审计证据。

### P0-3 规则生命周期模块

1. 目标：规则从“单版本编辑”升级为“可发布系统”。
2. 核心能力：草稿/发布版本、灰度范围、回滚、历史对比、事件回放验证。
3. DoD：规则上线前可回放验证，上线后支持一键回滚到指定版本。

### P0-4 工作流编排与执行引擎模块

1. 目标：从单动作执行升级为多步骤编排。
2. 核心能力：条件分支、串并行步骤、重试/补偿、人工审批节点。
3. DoD：至少支持「标签+评论+通知」三步流程，失败可重试并可补偿。

### P0-5 可观测性与 SRE 基线模块

1. 目标：建立可运营基线。
2. 核心能力：SLI/SLO、错误预算、多级告警、任务队列与执行延迟看板。
3. DoD：可在 5 分钟内定位 webhook 丢失、执行失败、队列堆积三类问题。

### P0-6 数据治理基线模块

1. 目标：控制数据成本与合规风险。
2. 核心能力：分级保留策略、归档、敏感字段脱敏、导出审计。
3. DoD：事件与审计数据可按策略自动过期/归档，策略变更可审计。

## 3.2 P1（增强，规模化能力）

### P1-1 多平台集成中心

1. 支持 GitLab/Gitea 事件接入。
2. 支持 Slack/邮件/IM 通知动作。
3. 统一连接器配置模型与健康检查。

### P1-2 通知与值班协同模块

1. 告警路由、升级策略、值班轮值。
2. 与执行失败记录联动，避免重复噪音通知。

### P1-3 分析与运营报表模块

1. 规则命中率、误报率、处理时长趋势。
2. 团队/仓库维度 ROI 指标（节省的人力处理时间）。

### P1-4 对外 API 与 SDK 模块

1. OpenAPI 文档与 Token 管理。
2. 官方 SDK（先 Go/TypeScript）与 webhook 回调订阅能力。

## 3.3 P2（扩展，平台生态）

### P2-1 插件与模板市场

1. 动作插件、规则模板、流程模板的安装与版本管理。
2. 插件签名校验与权限沙箱。

### P2-2 Policy as Code 模块

1. 规则与流程可声明式管理（GitOps）。
2. 审批、评审、回滚纳入代码变更流程。

### P2-3 智能辅助模块（可选）

1. 噪音事件聚类与优先级建议。
2. 保持“建议优先、人类确认”的安全边界。

---

## 4. 模块依赖关系（执行顺序）

```text
[租户边界 P0-1] -> [权限治理 P0-2] -> [规则生命周期 P0-3]
                                   -> [工作流编排 P0-4]

[工作流编排 P0-4] -> [通知值班 P1-2]
[审计与指标 P0-5] -> [分析报表 P1-3]
[数据治理 P0-6]   -> [对外 API/SDK P1-4]
[连接器框架 P1-1] -> [插件市场 P2-1]
[规则版本化 P0-3] -> [Policy as Code P2-2]
```

原则：先做边界与治理，再做规模化和生态扩展。

---

## 5. 里程碑建议（按阶段门禁）

### 阶段 A（4-6 周）：P0 安全与治理底座

1. 完成 P0-1、P0-2。
2. 门禁：租户隔离测试、授权测试、审计追踪测试全部通过。

### 阶段 B（4-6 周）：P0 决策与执行升级

1. 完成 P0-3、P0-4。
2. 门禁：规则回放准确率、流程成功率、失败补偿闭环可验证。

### 阶段 C（3-4 周）：P0 运维与数据治理

1. 完成 P0-5、P0-6。
2. 门禁：SLO 看板可用、告警降噪生效、数据保留策略自动执行。

### 阶段 D（6-8 周）：P1 规模化能力

1. 完成 P1-1 ~ P1-4（按业务价值排序上线）。
2. 门禁：新增连接器接入时不修改核心规则引擎代码。

---

## 6. 最小可执行路线（推荐）

1. 先执行 `P0-1 -> P0-2 -> P0-3`，锁定系统边界与变更安全性。
2. 再执行 `P0-4`，把当前动作能力升级为流程能力。
3. 最后补 `P0-5/P0-6`，确保系统可运维、可持续。

---

## 7. 总结

1. 作为 MVP，当前规划已完整并可运行。
2. 作为产品级系统，V2 需要以 P0 六大模块补齐治理与平台化能力。
3. 推荐按本文阶段门禁推进，避免“功能先行、治理滞后”的返工风险。
